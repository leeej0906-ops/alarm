# PRD: 알람 웹앱 — 최신 뉴스 검색 및 음성 읽기 기능

- 문서 버전: v0.1 (Draft)
- 작성일: 2026-07-24
- 관련 문서: [최근뉴스_기술명세서.md](../최근뉴스_기술명세서.md)

---

## 1. 배경 및 목표

### 배경

기존 알람 웹앱은 설정한 시간에 알람 소리를 울리는 단순한 기능만 제공한다. 사용자가 아침에 알람을 끄고 나서 바로 스마트폰이나 PC로 뉴스를 확인하는 행동 패턴이 흔한데, 이 과정을 알람 앱 안에서 바로 해결해주면 사용자 경험을 개선할 수 있다는 아이디어에서 출발한다.

이번 기능은 알람이 울릴 때(또는 알람 화면 진입 시) 최신 뉴스 5건을 자동으로 검색해서 보여주고, 사용자가 이를 음성(TTS)으로 들을 수 있도록 하는 것을 목표로 한다.

### 목표

- 알람이 울리는 시점에 최신 뉴스 5건을 자동으로 가져와 화면에 노출한다.
- 사용자가 개별 뉴스 또는 전체 뉴스를 음성으로 들을 수 있게 한다.
- 외부 뉴스 API 키(네이버 API Client ID/Secret 등 민감 정보)를 안전하게 관리하여 보안 사고를 예방한다.
- 뉴스 브리핑 기능의 성공 여부와 관계없이, 알람 앱 본연의 기능(정해진 시간에 소리로 사용자를 깨우는 것)은 항상 보장한다.

### 비목표(참고)

- 뉴스 큐레이션 품질 자체를 고도화하는 것(개인화 추천 등)은 이번 기능의 핵심 목표가 아니다.

---

## 2. 사용자 스토리

1. **아침 기상 사용자로서**, 알람이 울릴 때 최신 뉴스 5건이 화면에 자동으로 떠서, 알람을 끄자마자 오늘의 주요 뉴스를 확인하고 싶다.
2. **바쁜 아침을 보내는 사용자로서**, 화면을 보지 않고도 뉴스를 들을 수 있도록 개별 뉴스 또는 전체 뉴스 5건을 순서대로 음성으로 읽어주었으면 한다.
3. **눈이 피로하거나 화면을 보기 어려운 사용자로서**, 뉴스 제목과 요약을 텍스트뿐 아니라 음성으로도 제공받고 싶다.
4. **모든 사용자로서**, 뉴스 검색이나 음성 읽기 기능에 장애가 발생하더라도 알람 소리 자체는 반드시 정상적으로 울려서, 기상이라는 핵심 기능이 방해받지 않았으면 한다.
5. **서비스 운영자/개발자로서**, 뉴스 API 키와 같은 민감한 인증 정보가 클라이언트(브라우저)에 노출되지 않아, 키 탈취나 남용으로 인한 비용·보안 사고를 방지하고 싶다.

---

## 3. 기능 요구사항

### 3.1 뉴스 검색

- FR-1. 알람이 울리는 시점(또는 알람 화면 진입 시점)에 서버 측(백엔드/Edge Function)에서 최신 뉴스 검색 API를 호출하여 뉴스 5건을 가져온다.
- FR-2. 각 뉴스 항목은 최소한 제목, 요약(2줄 내외), 출처/링크 정보를 포함해야 한다.
- FR-3. 뉴스 검색에 사용할 키워드는 현재 확정되지 않았다. (자세한 내용은 [Open Questions](#6-open-questions) 참조)
- FR-4. 뉴스 5건을 화면에 카드 형태로 렌더링한다.
- FR-5. 뉴스 데이터는 DB에 저장하지 않고, 매 요청마다 실시간으로 외부 뉴스 API를 조회하여 응답한다. (자세한 내용은 [비기능 요구사항](#4-비기능-요구사항보안-최우선) 참조)

### 3.2 음성 읽기(TTS)

- FR-6. 사용자가 개별 뉴스 카드를 클릭/선택하면 해당 뉴스(제목+요약)를 음성으로 읽어준다.
- FR-7. 사용자가 "전체 읽기"를 실행하면 뉴스 5건을 순서대로 이어서 읽어준다.
- FR-8. 사용자가 언제든 음성 읽기를 중지할 수 있어야 한다.
- FR-9. 음성 읽기는 한국어(ko-KR)를 기본으로 지원한다.
- FR-10. 현재 음성 읽기 진행 상태(재생 중/중지됨)를 UI에 표시한다.

### 3.3 알람 연동

- FR-11. 뉴스 브리핑 기능은 알람의 부가 기능이며, 알람 소리 재생 로직과 독립적으로 동작해야 한다. (자세한 내용은 [비기능 요구사항](#4-비기능-요구사항보안-최우선) 참조)
- FR-12. 뉴스 검색/음성 읽기 실행 여부와 무관하게, 기존 알람 CRUD(추가/수정/삭제/on-off) 기능은 변경 없이 그대로 동작해야 한다.

## 4. 비기능 요구사항(보안 최우선)

### 4.1 보안 (최우선)

- NFR-1. **네이버 API 키(Client ID/Secret)는 프론트엔드(브라우저에 배포되는 코드, 번들, 네트워크 요청 등 어떤 경로로도)에 절대 노출되어서는 안 된다.**
  - 키는 반드시 서버 측(백엔드/Edge Function 등)의 환경 변수/시크릿 저장소에서만 관리한다.
  - 프론트엔드는 자체 백엔드 프록시(Edge Function 등)를 통해서만 뉴스 데이터를 요청하며, 외부 뉴스 API를 직접 호출하지 않는다.
  - 브라우저 개발자 도구의 네트워크 탭, 소스맵, 클라이언트 번들 코드 어디에서도 API 키 문자열이 확인되지 않아야 한다.
- NFR-2. API 키 등 민감 정보는 소스 코드 저장소(Git)에 커밋되지 않아야 하며, `.env` 등 환경 변수 파일은 `.gitignore`로 관리되어야 한다.
- NFR-3. 뉴스 검색 프록시 엔드포인트(백엔드)는 인증되지 않은 무제한 호출을 방지하기 위한 최소한의 오남용 방지 조치(예: rate limiting)를 고려해야 한다. (구체적인 방식은 Open Questions 참조)

### 4.2 데이터 처리

- NFR-4. **뉴스 데이터는 DB(영구 저장소)에 저장하지 않는다. 매 요청마다 외부 뉴스 API를 실시간으로 호출하여 응답한다.**
  - 캐시 테이블이나 캐싱 레이어를 두지 않는다. (기존 기술명세서의 `news_cache` 캐싱 설계는 본 요구사항과 상충하므로 재검토가 필요하다 — [Open Questions](#6-open-questions) 참조)

### 4.3 안정성/장애 격리

- NFR-5. **뉴스 브리핑 기능(뉴스 검색 실패, API 타임아웃, TTS 오류 등)이 실패하더라도 알람 소리 자체는 반드시 정상적으로 울려야 한다.**
  - 알람 소리 재생 로직은 뉴스 검색/음성 읽기 로직과 완전히 분리되어, 뉴스 관련 예외가 알람 트리거를 막거나 지연시키지 않아야 한다.
  - 뉴스 검색 실패 시에는 에러를 사용자에게 조용히 알리거나(예: "뉴스를 불러올 수 없습니다" 안내), 뉴스 영역만 비활성/숨김 처리하고 알람 UI/소리는 정상 노출한다.

### 4.4 성능

- NFR-6. 뉴스 검색은 알람이 울리는 시점의 사용자 경험을 저해하지 않도록 합리적인 시간 내(구체적 목표치는 Open Questions 참조)에 응답하거나 타임아웃 처리되어야 한다.

### 4.5 호환성

- NFR-7. TTS는 브라우저 내장 Web Speech API를 기본으로 하며, 이를 지원하지 않는 브라우저/환경에서는 텍스트(제목/요약)만 정상적으로 노출되어야 한다(기능 저하는 허용하되 전체 기능 중단은 없어야 한다).

## 5. 범위 제외(Out of Scope)

- 뉴스 개인화 추천(사용자별 관심사 학습 기반 추천)
- 뉴스 검색 결과에 대한 사용자 피드백/평가(좋아요, 신고 등) 기능
- 뉴스 원문 전체 크롤링 및 본문 전체 표시(요약 수준까지만 제공)
- 다국어 뉴스/다국어 TTS 지원(우선 한국어만 지원)
- 뉴스 데이터의 영구 보관, 검색 이력 조회, 통계/분석 기능
- Google Cloud TTS, CLOVA Voice 등 고급 TTS 엔진 도입(1차 범위는 Web Speech API로 한정)
- 알람 다중 기기 실시간 동기화(Supabase Realtime 연동)는 본 기능과 별개 사안으로 취급

## 6. Open Questions

- **Q1. 뉴스 검색 키워드가 아직 확정되지 않았다.** 고정 키워드("주요 뉴스" 등 특정 카테고리)를 사용할지, 사용자가 직접 키워드를 입력/선택하게 할지, 혹은 알람별로 다른 키워드를 설정할 수 있게 할지 결정이 필요하다.
- **Q2. 뉴스 API 공급자가 명확하지 않다.** 사용자 요구사항은 "네이버 API(Client ID/Secret)"를 명시하지만, 기존 기술명세서(`최근뉴스_기술명세서.md`)는 Claude API + Web Search 또는 NewsAPI.org를 언급하고 있어 상충한다. 최종 채택할 뉴스 API를 확정하고 기술명세서를 업데이트해야 한다.
- **Q3. `news_cache` 테이블/캐싱 설계와의 충돌.** 기존 기술명세서는 1시간 캐시 테이블(`news_cache`)을 전제로 하지만, 본 PRD는 "DB에 저장하지 않고 매 요청 실시간 조회"를 요구한다. 캐싱을 완전히 제거할지, 혹은 캐싱이 필요한 경우 DB가 아닌 다른 방식(메모리 캐시 등)으로 한정할지 결정이 필요하다.
- **Q4. 뉴스 브리핑 트리거 시점.** 알람이 "울리는 순간"에 뉴스를 검색할지, 알람을 끈 직후에 검색할지, 혹은 별도의 사용자 액션(버튼 클릭)으로 검색을 시작할지 확정이 필요하다.
- **Q5. 응답 시간/타임아웃 기준값.** 뉴스 검색 API 응답 지연에 대한 구체적인 타임아웃 기준(예: 3초, 5초)이 아직 정해지지 않았다.
- **Q6. Rate limiting 구체안.** 백엔드 프록시에 대한 오남용 방지(rate limiting)를 어떤 단위(사용자별/IP별/전역)로, 어느 정도 수준으로 적용할지 결정이 필요하다.
- **Q7. 인증되지 않은 사용자 처리.** 로그인하지 않은 사용자도 뉴스 브리핑 기능을 사용할 수 있는지, 아니면 로그인 사용자에게만 제공할지 결정이 필요하다.

## 7. 성공 기준

- 알람이 울릴 때 최신 뉴스 5건이 예외 없이 화면에 표시된다(뉴스 API 정상 응답 시).
- 네이버 API 키(Client ID/Secret)가 프론트엔드 번들, 네트워크 요청, 소스코드 저장소 어디에도 노출되지 않음을 코드 리뷰 및 네트워크 점검으로 확인한다.
- 뉴스 데이터가 DB 등 영구 저장소에 저장되지 않고 매 요청마다 실시간으로 조회됨을 확인한다(저장 로직/테이블 부재를 코드로 검증).
- 뉴스 검색 API 또는 TTS 기능에 강제로 장애(타임아웃, 오류 응답 등)를 주입했을 때도 알람 소리가 설정된 시간에 정상적으로 울리는 것을 테스트로 확인한다.
- 사용자가 개별 뉴스 및 전체 뉴스 5건을 음성으로 정상적으로 들을 수 있고, 언제든 중지할 수 있다.
- 위 [Open Questions](#6-open-questions) 항목들이 이해관계자와의 논의를 통해 확정되어, 후속 설계/개발 단계로 넘어갈 수 있는 상태가 된다.
