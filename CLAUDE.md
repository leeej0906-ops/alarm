# CLAUDE.md — 프로젝트 공통 규칙

이 문서는 이 프로젝트 전체에서 Claude Code가 항상 지켜야 할 공통 규칙을 담는다.
뉴스 기능만의 세부 규칙(파일 구조, 함수 이름 등)은 여기에 포함하지 않는다.

---

## 🚨 최우선 보안 규칙: 네이버 API 키는 프론트엔드에 절대 넣지 않는다

**네이버 API 키(Client ID/Secret)는 어떤 경로로도 프론트엔드(브라우저에 배포되는 코드)에 노출되어서는 안 된다.**

- 브라우저 번들, 소스맵, 네트워크 요청, 클라이언트 측 환경 변수(`NEXT_PUBLIC_*`, `VITE_*` 등 프론트엔드에 인라인되는 접두사) 어디에도 키 문자열이 포함되어서는 안 된다.
- 키는 반드시 서버 측(백엔드/Edge Function)의 환경 변수·시크릿 저장소에서만 관리한다.
- 프론트엔드는 자체 백엔드 프록시를 통해서만 뉴스 데이터를 요청하며, 외부 뉴스 API를 직접 호출하지 않는다.
- 이 규칙을 위반하는 코드는 어떤 이유로도 작성하거나 커밋하지 않는다.

---

## 보안 및 민감 정보

- API 키 등 민감 정보는 Git 저장소에 커밋하지 않는다. `.env` 등 환경 변수 파일은 반드시 `.gitignore`로 관리한다.
- 인증되지 않은 무제한 호출을 방지하기 위해, 외부 API를 호출하는 백엔드 프록시 엔드포인트에는 최소한의 오남용 방지 조치(rate limiting 등)를 고려한다. 구체적인 방식은 아직 확정되지 않았다(아래 "미확정 사항" 참고).

## 데이터 처리 원칙

- 외부에서 실시간으로 조회하는 데이터(예: 뉴스)는 DB(영구 저장소)에 저장하지 않는다. 매 요청마다 실시간으로 외부 API를 호출하여 응답하며, 캐시 테이블이나 캐싱 레이어를 두지 않는다.

## 실패 허용 원칙 (Fail-safe)

- **알람 소리 자체는 부가 기능의 성공 여부와 무관하게 반드시 울려야 한다.** 뉴스 브리핑 등 알람의 부가 기능(검색 실패, API 타임아웃, TTS 오류 등)이 실패하더라도 알람 트리거를 막거나 지연시켜서는 안 된다.
- 알람 소리 재생 로직은 부가 기능 로직과 완전히 분리하여 구현한다. 부가 기능에서 발생한 예외가 알람 재생 로직으로 전파되지 않도록 한다.
- 부가 기능 실패 시에는 조용히 처리한다(에러를 사용자에게 부드럽게 안내하거나 해당 영역만 비활성/숨김 처리). 알람 UI/소리는 항상 정상 노출한다.

## 호환성 원칙

- 브라우저/환경별로 지원되지 않는 기능(예: Web Speech API 미지원)이 있을 경우, 기능 저하(텍스트만 노출 등)는 허용하되 전체 기능이 중단되어서는 안 된다.

## 작업 순서 원칙

새로운 기능을 구현할 때는 다음 순서를 지킨다.

1. 먼저 화면(UI)을 가짜 데이터(mock data)로 만들어 동작을 확인한다.
2. 그 다음 실제 백엔드를 연결한다.
3. 예외 처리(에러 핸들링, 타임아웃, 장애 격리 등)는 가장 마지막에 추가한다.

이 순서를 임의로 바꾸거나 건너뛰지 않는다.

## 미확정 사항 (Open Questions) — 임의로 결정하지 말 것

다음 사항들은 아직 이해관계자와 확정되지 않았다. 관련 작업을 할 때 임의로 결정하지 말고, 확정이 필요하면 사용자에게 먼저 질문할 것.

- 뉴스 검색 키워드 방식(고정 키워드 / 사용자 입력 / 알람별 설정 등)
- 뉴스 브리핑 트리거 시점(알람이 울리는 순간 / 알람을 끈 직후 / 별도 사용자 액션 등)
- 그 외 뉴스 API 공급자, 캐싱 설계, 응답 시간/타임아웃 기준값, rate limiting 구체안, 인증되지 않은 사용자 처리 등 세부 사항
